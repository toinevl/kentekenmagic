Card component patterns from LookupExperience.tsx (reference existing cards for style conventions):
- Null guard: if (!prop) return null
- Card wrapper: article element with Tailwind card classes (border, rounded-lg, shadow-sm, p-5)
- Section heading: h2 or similar with icon from lucide-react
- Detail list: dl with dt/dd pairs (from TechCard/RegistrationCard pattern)
- List items: ul with li (from FuelCard pattern)
- Status badge: inline pill span with color-coded Tailwind classes

RecallCard design decisions (planner discretion per CONTEXT.md):
- Shown as a separate card after ApkTimelineCard (not integrated into it)
- If recallStatus is undefined: return null (no card rendered — don't show "Geen gegevens")
- If hasOpenRecall: false and recalls empty: render green "Geen terugroepactie" single-line card
- If hasOpenRecall: true: render red badge + list of recall entries with defectDescription, riskLevel, repairDescription
- riskLevel display: "ERN" → "Hoog risico", "MID" → "Gemiddeld risico", "LOW" → "Laag risico"; fallback to raw value

ModificationsCard design decisions (planner discretion per CONTEXT.md):
- Only show card when modifications is defined (if undefined: return null)
- Show card even when modifications array is empty (empty state: "Geen geregistreerde wijzigingen")
- Each modification row: description + installDate + "Actief" green badge if isActive, "Verwijderd" gray badge if not
- Include manufacturer and tankCapacity when present (e.g., "LPG · 60L" suffix)
- Card position: after FuelCard, before RegistrationCard

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| API response → frontend card props | cards.rdw_recall_status and cards.rdw_modifications cast from unknown; no frontend Zod validation |
| Recall description text → DOM | omschrijving_defect, beschrijving_van_het_herstel rendered as React text nodes |
| External URL from recall | meer_informatie_op_internet field; if rendered as anchor, must have rel="noopener noreferrer" |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-05-01 | Tampering | RecallCard text fields from RDW (defectDescription, repairDescription) | accept | React auto-escapes text node content; no dangerouslySetInnerHTML; RDW data is official government source |
| T-03-05-02 | Tampering | moreInfoUrl from RDW rendered as anchor | mitigate | If moreInfoUrl is rendered as an anchor tag, add rel="noopener noreferrer" and target="_blank"; do not render as href if value contains "javascript:" (unreachable from RDW but defensive) |
| T-03-05-03 | Tampering | ModificationsCard description text from RDW | accept | Rendered as React text node; no script injection surface |
| T-03-05-04 | Information Disclosure | Empty-state card reveals source structure | accept | "Geen terugroepactie" and "Geen geregistreerde wijzigingen" reveal nothing about internal APIs or data structure |
| T-03-05-05 | Denial of Service | Rendering large modifications array | accept | RDW sghb-dzxx results are bounded by registered modifications per vehicle; no pagination needed |
| T-03-05-SC | Tampering | npm/pip/cargo installs | accept | No new packages in this plan |
</threat_model>
