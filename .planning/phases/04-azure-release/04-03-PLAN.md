    Write DEPLOY.md at the repo root as a practical provisioning runbook. Structure:

    ## Prerequisites
    List: Azure CLI installed and logged in (`az login`), GitHub repo access with Settings > Secrets write permission, Socrata RDW app token from data.overheid.nl, Anthropic API key from console.anthropic.com.

    ## Step 1: Create Resource Group
    Command: az group create --name kentekenmagic-rg --location westeurope

    ## Step 2: Create Static Web Apps Resource
    Command: az staticwebapp create --name kentekenmagic --resource-group kentekenmagic-rg --location westeurope --sku Free
    Note: Do NOT link to GitHub via Portal — linking via portal auto-generates a workflow with a different secret name (Pitfall 5 from research).

    ## Step 3: Get Deployment Token
    Command: az staticwebapp secrets list --name kentekenmagic --query "properties.apiKey" -o tsv
    Instruction: Copy the output. Go to GitHub repo Settings > Secrets and variables > Actions > New repository secret. Name: AZURE_STATIC_WEB_APPS_API_TOKEN. Value: the token.

    ## Step 4: Create Storage Account
    Note on name uniqueness: Storage account names are globally unique, 3-24 chars, lowercase only. If kentekenmagicstorage is taken, append a short suffix.
    Command: az storage account create --name kentekenmagicstorage --resource-group kentekenmagic-rg --location westeurope --sku Standard_LRS --kind StorageV2

    ## Step 5: Create Cache Tables
    Commands using shell variable pattern:
    CONN_STR=$(az storage account show-connection-string --name kentekenmagicstorage --resource-group kentekenmagic-rg -o tsv)
    az storage table create --name VehicleCache --connection-string "$CONN_STR"
    az storage table create --name LlmSummaryCache --connection-string "$CONN_STR"

    ## Step 6: Set Application Settings
    az staticwebapp appsettings set --name kentekenmagic --resource-group kentekenmagic-rg --setting-names "FUNCTIONS_WORKER_RUNTIME=node" "AZURE_STORAGE_CONNECTION_STRING=$CONN_STR" "RDW_APP_TOKEN=<your-socrata-token>" "ANTHROPIC_API_KEY=<your-anthropic-key>"
    Note: Replace <your-socrata-token> and <your-anthropic-key> with real values. OPENAI_API_KEY is intentionally omitted (Anthropic only, per D-08).

    ## Step 7: Trigger First Deploy
    Option A: Push any commit to main. Option B: GitHub repo > Actions > "Deploy to Azure Static Web Apps" > Run workflow.
    Monitor progress in GitHub Actions tab.

    ## Step 8: Verify Deployment
    Check SWA hostname: az staticwebapp show --name kentekenmagic --query "defaultHostname" -o tsv
    Navigate to https://<defaultHostname> in a browser.

    ## Smoke Test (after Step 7)
    See the smoke test section of this runbook (also enforced by Plan 03 checkpoint task).

    ## Troubleshooting
    Include abbreviated versions of Pitfalls 1-5 from RESEARCH.md as a reference table:
    - Pitfall 1: staticwebapp.config.json not in out/ — fix: confirm it's at frontend/public/
    - Pitfall 2: TypeScript not compiled — fix: confirm "Build API" step ran in deploy.yml
    - Pitfall 3: Storage connection string not set — fix: re-run Step 6
    - Pitfall 4: output_location path confusion — fix: if blank page, change output_location to "frontend/out"
    - Pitfall 5: Secret name mismatch — fix: do not use Portal-linked workflow; use manually created AZURE_STATIC_WEB_APPS_API_TOKEN secret

    ## Application Insights (optional)
    Set APPLICATIONINSIGHTS_CONNECTION_STRING via az staticwebapp appsettings set (same command as Step 6, add the setting). KQL query for cold start measurement is documented in 04-RESEARCH.md.

    Keep the document concise. All CLI commands must be in bash code blocks. Include verify commands after each step where possible (e.g., az staticwebapp show after Step 2, az storage table list after Step 5).

| Boundary | Description |
|----------|-------------|
| DEPLOY.md content → production environment | Runbook commands write secrets as application settings; secrets must never appear in DEPLOY.md itself — only placeholder tokens |
| Production HTTPS endpoint → public internet | SWA handles TLS; no additional cert management needed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-08 | Information Disclosure | DEPLOY.md | mitigate | All CLI commands use placeholder values for secrets (your-anthropic-key, your-socrata-token); acceptance criteria verify no real keys present |
| T-04-09 | Information Disclosure | Production application settings | mitigate | Secrets set only via az staticwebapp appsettings set; never written to files; DEPLOY.md acceptance criteria verify no AccountKey= or sk-ant patterns |
| T-04-10 | Denial of Service | RDW API dependency | accept | RDW is a government open API; no rate-limit issues expected at personal-project scale; tableCache reduces RDW calls |
| T-04-11 | Spoofing | Unauthenticated /api/* endpoints | accept | RDW vehicle data is public; no per-user data; no auth required for v1 — documented decision |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed in this plan |
</threat_model>
