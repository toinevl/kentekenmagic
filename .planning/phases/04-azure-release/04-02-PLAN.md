<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → Azure SWA | Deployment token authorizes uploads; must never appear in source |
| PR source → validate.yml | Untrusted PR code runs lint/typecheck/build; no secrets accessible |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04 | Information Disclosure | .github/workflows/deploy.yml | mitigate | AZURE_STATIC_WEB_APPS_API_TOKEN referenced only via ${{ secrets.* }} — acceptance criteria require this; any literal token value fails the check |
| T-04-05 | Tampering | GitHub Actions workflow trigger | accept | No approval gate needed — personal project; PRs go through validate.yml first |
| T-04-06 | Information Disclosure | validate.yml env vars | mitigate | AZURE_STORAGE_CONNECTION_STRING set to UseDevelopmentStorage=true in validate.yml (not real connection string); no real secrets in workflow files |
| T-04-07 | Spoofing | SWA deployment token rotation | accept | Token rotation is managed by Azure; `az staticwebapp secrets list` can re-fetch if needed; documented in DEPLOY.md |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed in this plan; all actions are official Microsoft/GitHub-verified actions |
</threat_model>
